什么是 HarnSpec？

HarnSpec 是一种轻量级方法论，用于编写人类和 AI 代理都能执行的 Spec。它专注于捕获 意图、约束 和 成功标准，而不让读者陷入繁文缛节。

时间紧张？

浏览标注为 为什么重要 的部分，在深入了解之前先理解每个概念的目的。

为什么需要 HarnSpec​

传统 Spec 无法满足新约束​

  • 太长 → 超出人类/AI 工作记忆 (Working Memory)（违反上下文经济 (Context Economy) 原则）
  • 太详细 → 快速过时，变得不可信
  • 太死板 → 在需要迭代时阻碍进展
  • 太模糊 → AI 无法执行，人类不断提出后续问题

AI 驱动开发带来的差距​

人类以目标思考（"让仪表板更快"）。AI 代理需要明确的护栏（"在 Redis 中缓存数据，目标 <100ms p95"）。HarnSpec 通过结构化但可读的文档弥合这一差距。

HarnSpec 原则（入门版）​

  1. 上下文经济 (Context Economy) — 将每个文件保持在约 2,000 Token 以内，以适应工作记忆 (Working Memory)。
  2. 信噪比 (Signal-to-Noise) — 每句话都应该为决策提供信息。
  3. 意图优于实现 (Intent Over Implementation) — 捕获为什么和是什么；让如何做在执行过程中出现。
  4. 弥合差距 (Bridge the Gap) — 编写让人类和 AI 都能理解相同目标的文档。
  5. 渐进式披露 (Progressive Disclosure) — 从简单开始；在出现问题时添加子 Spec 或字段。

你不必在第一天就记住这些。当 Spec 感觉臃肿或不清楚时，将它们用作检查清单。

HarnSpec 解决什么问题？​

  • 人类和 AI 助手之间 缺乏共享上下文
  • 聊天记录消失时 决策丢失
  • 没有人愿意维护的 过度膨胀文档
  • 所有权不清，因为状态和元数据分散在其他地方

HarnSpec 的答案：保持 Spec 简短、有目的性和结构化。CLI 和 Agent Skills 使它们易于搜索、验证并交给 AI 代理。

何时编写 Spec​

在以下情况下编写或更新 Spec：

  • 意图需要澄清或存在多种解释
  • 权衡、约束或成功标准很重要
  • 工作跨越多个文件/系统或影响其他团队
  • AI 代理将实现功能的一部分
  • 决策应该在聊天或会议之外持续存在

何时跳过​

在以下情况下跳过（或推迟）Spec：

  • 修复明显的 bug，且修复方法明确
  • 执行机械式重构（重命名、格式化、依赖升级）
  • 原型设计以学习——如果它转为实际工作，则捕获结果

HarnSpec + AI​

  • 创作：提供意图；让 AI 起草结构；你编辑以确保清晰。
  • 实现：将 Spec 提供给 AI，以便它可以自主规划和修改代码。
  • 记忆：Spec 作为持久知识。代理在重新参与时通过 Agent Skills 重新加载它们。
